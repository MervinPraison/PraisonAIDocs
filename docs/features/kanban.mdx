## Task dependencies & auto-promotion

Link tasks with parent edges using `kanban_link` — when all parents complete, the child task automatically becomes `ready` on the next dispatcher tick. No flag or manual call needed.

```mermaid
graph LR
    Parent1[✅ Parent 1] --> Child[⏳ Child task]
    Parent2[✅ Parent 2] --> Child
    Child -.auto-promote.-> Ready[🎯 Ready]

    classDef done fill:#10B981,stroke:#7C90A0,color:#fff
    classDef pending fill:#F59E0B,stroke:#7C90A0,color:#fff
    classDef ready fill:#189AB4,stroke:#7C90A0,color:#fff

    class Parent1,Parent2 done
    class Child pending
    class Ready ready
```

```bash
praisonai kanban create "Design auth flow" --status done   # parent 1
praisonai kanban create "Set up DB schema" --status done   # parent 2
praisonai kanban create "Implement auth API" --status todo  # child

praisonai kanban link design_task auth_task
praisonai kanban link db_task auth_task

# On the next dispatcher tick, auth_task moves to ready automatically.
```

<Note>
**Parentless `todo` tasks are intentionally left untouched.** Only tasks with at least one parent link participate in auto-promotion. `todo` remains a valid manual staging column for tasks you want to promote yourself.
</Note>

Each auto-promoted task fires a `KANBAN_TASK_MOVED` event. See `kanban_link` in the [Agent Tools](#agent-tools) section for how to create parent edges.

## How It Works

```mermaid
sequenceDiagram
    participant User
    participant Coordinator
    participant Store
    participant Dispatcher
    participant Worker
    participant UI
    
    User->>Coordinator: "Build login system"
    Coordinator->>Store: Create tasks (design, implement, test)
    Store->>Dispatcher: Dispatch ready tasks
    Dispatcher->>Worker: Claim task
    Worker->>Store: Add comments/heartbeat
    Worker->>Store: Mark completed
    Store->>UI: Update status
    UI->>User: Show progress
```

Task coordination happens through a SQLite-backed persistent store that all agents and the UI share.

| Component | Purpose |
|-----------|---------|
| **Kanban Store** | SQLite database storing tasks, comments, links |
| **Agent Tools** | 8 functions for task CRUD operations |
| **CLI Commands** | Human interface for task management |
| **Background Dispatcher** | Auto-claims ready tasks for processing |

## Reliability: claim leases & automatic reclamation

If a worker crashes mid-task, PraisonAI automatically retries by detecting the dead worker and putting the task back on the board. No manual intervention required.

```mermaid
sequenceDiagram
    participant D as Dispatcher
    participant W as Worker
    participant S as Store

    D->>S: claim_task(ttl=900s, pid=W.pid)
    S-->>D: task claimed (running)
    D->>W: spawn worker
    W->>S: heartbeat every ~60s
    Note over W: Worker crashes
    D->>S: reclaim_stale_claims() (next tick)
    Note over S: lease expired + PID dead
    S-->>D: task → ready
    D->>W: re-dispatch to new worker

    classDef dispatcher fill:#8B0000,stroke:#7C90A0,color:#fff
    classDef worker fill:#F59E0B,stroke:#7C90A0,color:#fff
    classDef store fill:#189AB4,stroke:#7C90A0,color:#fff

    class D dispatcher
    class W worker
    class S store
```

**Env vars for tuning:**

| Variable | Default | What it controls |
|----------|---------|-----------------|
| `PRAISONAI_KANBAN_CLAIM_TTL` | `900` (15 min) | Claim lease duration in seconds. After this, the claim is reclaimable if the worker is dead or stale. |
| `PRAISONAI_KANBAN_STALE_TIMEOUT` | `1800` (30 min) | Heartbeat staleness threshold. A live worker whose heartbeat is older than this is treated as hung. |

For long-running tasks, increase `PRAISONAI_KANBAN_CLAIM_TTL` to match the expected task duration. A live worker can extend its own lease by heartbeating — the reclaim sweep leaves it untouched.

The `KANBAN_TASK_RECLAIMED` event fires when a stranded task returns to `ready`, so monitoring hooks can react. The SQLite store auto-migrates pre-existing databases — no manual action needed on upgrade.

### Advanced: implementing a custom store

Custom store implementations can add lease + heartbeat support by conforming to `KanbanReclaimProtocol` in `praisonaiagents.kanban`. This is an additive extension protocol — existing `KanbanStoreProtocol` implementations are not broken.

Tasks form directed acyclic graphs through parent-child dependencies:

### Claim/Release

Workers coordinate through atomic claim and release operations:

```mermaid
sequenceDiagram
    participant Worker1
    participant Store
    participant Worker2
    
    Worker1->>Store: Claim ready task
    Store-->>Worker1: Task claimed (status: running)
    Worker2->>Store: Try claim same task
    Store-->>Worker2: Already claimed
    Worker1->>Store: Heartbeat signal
    Worker1->>Store: Release (complete/block)
    Store-->>Worker1: Task updated
```

## Agent Tools

Kanban tools are available through the SDK protocols. The wrapper implementation provides these agent tools:

### Task Management

| Tool | Purpose | Example |
|------|---------|---------|
| `kanban_create` | Create new task | `kanban_create("Implement auth", assignee="dev")` |
| `kanban_list` | Filter tasks | `kanban_list(status="ready", assignee="dev")` |
| `kanban_show` | Get task details | `kanban_show("task_abc123")` |

### Status Changes

| Tool | Purpose | Example |
|------|---------|---------|
| `kanban_complete` | Mark task done | `kanban_complete("task_abc123", "Auth working")` |
| `kanban_block` | Block with reason | `kanban_block("task_abc123", "Need API keys")` |

### Coordination

| Tool | Purpose | Example |
|------|---------|---------|
| `kanban_comment` | Add progress note | `kanban_comment("task_abc123", "50% complete")` |
| `kanban_link` | Create dependency | `kanban_link("design_task", "implement_task")` |
| `kanban_heartbeat` | Signal liveness | `kanban_heartbeat("task_abc123", "testing")` |

## Best Practices

<AccordionGroup>
<Accordion title="Task Granularity">
Create tasks that can be completed in 15-30 minutes. Break larger work into linked subtasks using `kanban_link` for proper dependency tracking.
</Accordion>

<Accordion title="Status Management">
Move tasks through statuses systematically: `todo` → `ready` → `running` → `done`. Use `blocked` for dependencies and `review` for human approval.
</Accordion>

<Accordion title="Agent Coordination">
Use `kanban_heartbeat` during long-running tasks to signal liveness. Add detailed comments with `kanban_comment` to track progress and decisions.
</Accordion>

<Accordion title="Board Organization">
Use separate boards for different projects or contexts. Default board works well for single-project setups.
</Accordion>
</AccordionGroup>
