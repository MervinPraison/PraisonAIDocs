PraisonAI auto-migrates legacy `bot.yaml` and BotOS `platforms:` configs to the canonical `GatewayConfigSchema` at load time ‚Äî `praisonai doctor` reports migration opportunities so you can persist them.

```bash
praisonai doctor --only gateway_config_migration
```

## Quick Start

<Steps>

<Step title="Detect legacy format">

```bash
praisonai doctor --only gateway_config_migration
```

If your config is already canonical, you will see **PASS ‚Äî Config uses current format**.

</Step>

<Step title="Review WARN output">

Example when migration is available:

```
‚öÝ Gateway Config Migration [MEDIUM]
  Config can be migrated: 2 change(s)
  ‚Ä¢ Single-bot format can be migrated to multi-channel format
  ‚Ä¢ telegram: allowed_users string ‚Üí list migration available
```

</Step>

<Step title="Persist canonical YAML">

Rewrite your config to the canonical `channels:` form (see migration table below). At runtime, legacy shapes already load ‚Äî persisting is optional but recommended for clarity.

</Step>

</Steps>

## Migration Table

### Single-bot ‚Üí multi-channel

**Before** (legacy `bot.yaml`):

```yaml
platform: telegram
token: ${TELEGRAM_BOT_TOKEN}
agent:
  name: assistant
  instructions: "Help users"
```

**After** (canonical):

```yaml
agent:
  name: assistant
  instructions: "Help users"
channels:
  telegram:
    platform: telegram
    token: ${TELEGRAM_BOT_TOKEN}
```

### BotOS `platforms:` ‚Üí `channels:`

**Before**:

```yaml
agent:
  name: assistant
platforms:
  telegram:
    token: ${TELEGRAM_BOT_TOKEN}
  discord:
    token: ${DISCORD_BOT_TOKEN}
```

**After**:

```yaml
agent:
  name: assistant
channels:
  telegram:
    token: ${TELEGRAM_BOT_TOKEN}
  discord:
    token: ${DISCORD_BOT_TOKEN}
```

### String `allowed_users` ‚Üí list

**Before**:

```yaml
channels:
  telegram:
    token: ${TELEGRAM_BOT_TOKEN}
    allowed_users: "123456,789012"
```

**After**:

```yaml
channels:
  telegram:
    token: ${TELEGRAM_BOT_TOKEN}
    allowed_users:
      - "123456"
      - "789012"
```

## Behaviour Notes

- `BotYamlSchema` is an alias of `GatewayConfigSchema` ‚Äî existing Python imports keep working.
- `group_policy` defaults to `mention_only` for **new** channels without an explicit value. Configs that explicitly set `respond_all` keep that value.
- Comma-separated `allowed_users` strings are auto-converted to lists at load time.
- All three YAML shapes (`platform`+`token`, `agents`+`channels`, `platforms:`) validate against one schema ‚Äî see [Gateway](/docs/features/gateway).

## Strict Validation (PR #2439)

As of PR #2439, the following previously-silent conditions are now **fatal configuration errors** (exit code `78`). The gateway will not start and will not be restarted by the supervisor until the config is fixed:

- **`gateway.yaml` empty, no `agents` section, or no `channels` section** ‚Äî previously a silent `Warning`; now fatal at startup.
- **Missing token in any channel** ‚Äî previously ignored; now fatal.
- **`agents` section that is not a list of mappings** ‚Äî e.g. `agents: ["bad"]` or `agents: {name: bad}` is now fatal.
- **`--agents` file missing, unreadable, or malformed** ‚Äî the gateway no longer starts with an empty agent set; now fatal.
- **Invalid admission-control values** ‚Äî `max_concurrent_runs`, `queue_depth`, or `overflow_policy` values that fail schema validation are now fatal.
- **Missing `pip install praisonai[api]`** ‚Äî optional API dependencies are now detected at startup and map to exit `78` instead of a silent return.

<Warning>
If you upgrade and your gateway previously started with warnings about missing config sections, it will now exit immediately. Run `praisonai doctor --only gateway_config_migration` to check your config before upgrading.
</Warning>

See [Gateway Exit Codes](/docs/features/gateway-exit-codes) for how to configure your process supervisor to handle these exit codes.

## Best Practices

<AccordionGroup>
<Accordion title="Run doctor before upgrading">
Use `praisonai doctor --only gateway_config_migration` after pulling a new PraisonAI release to see whether your persisted YAML can be simplified.
</Accordion>

<Accordion title="Persist canonical YAML when WARN appears">
Runtime auto-migration is transparent, but persisting the canonical `channels:` form makes configs easier to diff and review in PRs.
</Accordion>

<Accordion title="Keep explicit group_policy values">
`group_policy` defaults to `mention_only` for new channels only. If your bot should respond to every message, set `respond_all` explicitly rather than relying on legacy defaults.
</Accordion>

<Accordion title="Convert allowed_users to lists">
Comma-separated strings still load, but list form is clearer and matches the schema validators.
</Accordion>
</AccordionGroup>
